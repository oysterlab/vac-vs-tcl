문제 정의

실행형 AI(Agent)는 이미
플랫폼 경쟁으로 이동하고 있으며,
주요 플레이어들은 이를 자사 생태계에
빠르게 편입시키고 있습니다.

하지만 우리는 아직 이 구조에서
어떤 레이어를 가져갈지 명확하지 않습니다.

03 / 12

시장에서 벌어지고 있는 일

OpenClaw는 실행형 AI를 처음으로
직접 체감할 수 있게 만든 기준점

AI가 단순 답변이 아니라, 실제 작업을 수행하는 구조.
Agent Runtime이라는 개념을 시장에 각인시켰다.

하지만 본질은 사용자가 직접 운영하는 self-hosted 구조.
설치 필요. API key 필요. 사용자 환경에 종속.

인지와 기대는 빠르게 확산됐지만,
실제 사용은 제한적
04 / 12

C-Brand 전략

Huawei, Xiaomi는
완전히 다른 방향을 선택

  1. 01

    Agent를 "시스템 레이어"로 내재화

    OS, 클라우드, 디바이스 전반에 통합

  2. 02

    OpenClaw 구조를 기반으로 "제품화"

    Agent runtime + tool + memory 구조 차용. 설치 없이 즉시 사용

  3. 03

    모델 선택 제거

    사용자에게 API / 모델 노출 안 함. 내부적으로 모델 routing

  4. 04

    데이터와 경험을 자사 계정에 축적

    행동, intent, 사용 패턴 전부 플랫폼이 수집

08 / 12

비용 구조

동일한 접근은
비용 구조가 맞지 않음

C-Brand는 모델 비용 + 실행 비용 전부를 플랫폼이 부담.
이것이 가능한 이유:

  1. C-Brand

    자체 모델(소형 LLM) 보유 + 클라우드 인프라 내재화

    Huawei: Pangu 모델 + Huawei Cloud. Xiaomi: MiLM + 자체 서버. 모델 운영 비용을 자사 인프라로 흡수

  2. VD

    자체 LLM 없음. 외부 모델 의존 시 비용이 사용량에 비례

    TV 사용자 수 x API 호출 비용 = 통제 불가능한 비용 구조. 동일한 방식은 성립하지 않음

구조를 바꿔야 함.
비용 구조가 다르면, 전략도 달라야 한다.
09 / 12

전략 제안

BYO AI — Bring Your Own AI

사용자가 자신의 AI 모델(ChatGPT, Claude, Gemini 등)을
API key를 통해 TV에 직접 연결하는 구조.
VD는 모델을 제공하지 않고, Agent 실행 환경만 제공.

모델은 사용자,
Agent 경험과 데이터는 VD

VD Agent Runtime · Memory · UX
User AI 모델 선택 · API key · LLM 비용
모델을 소유하지 않아도,
Agent 플랫폼은 확보 가능

토론

우리가 결정해야 할 것

  1. Q1

    VD가 소유해야 할 레이어는 어디까지인가?

    Agent Runtime만? Memory까지? UX 전체? 어디에 경계를 그을 것인가

  2. Q2

    BYO AI 온보딩을 어떻게 설계할 것인가?

    사용자에게 API key 입력을 요구하는 것은 높은 허들. TV 리모컨 환경에서 어떻게 자연스럽게 연결시킬 수 있는가

  3. Q3

    킬러 시나리오는 무엇인가?

    사용자가 BYO AI를 처음 연결했을 때, "이것 때문에 연결할 가치가 있다"고 느낄 첫 경험은 무엇이어야 하는가

  4. Q4

    데이터 축적 전략은?

    사용자 intent, 행동 패턴, Memory를 어떤 형태로 축적하고 활용할 것인가
